# 项目定义与边界

## “因时而得谷”是什么

“因时而得谷”（简称“因时”，英文仓库名 `Yinshi`）是一个个人设备远程协作与执行项目。它把飞书当作可靠的移动消息入口，把每台受控计算机上的本地 Codex 当作实际执行者。

典型场景：

1. 用户离开电脑后在手机飞书中向某台机器的 Codex 机器人发送任务。
2. 该机器上的桥服务接收消息，确认发件人是已配对的所有者。
3. 桥把任务路由到正确的本地 Codex thread，或者创建一个无上下文的临时任务。
4. Codex 在机器本地读取、修改、运行和验证。
5. 桥持续把确认、中间进度、审批、最终答复和交付文件回传飞书。

“因时而得谷”不仅指一段机器人代码，也包含：

- 飞书组织和每台机器的企业自建应用
- 机器上的 Codex CLI 与 App Server
- 本仓库中的桥接服务
- 配置、配对、运维、安全和灾难恢复流程
- 未来面向更多设备和开源参与者的工程规范

## 为什么选择飞书

- 中国大陆手机和桌面端可用。
- 支持私聊、群聊、交互卡片、图片、文件、音频和视频。
- 企业自建应用可通过长连接接收事件，不需要给家庭/办公电脑开放公网端口。
- 移动端通知、桌面端历史记录和多设备登录由飞书承担。

## 会话模型

每台机器只使用一个机器人应用，例如：

```text
Codex-鼎盛笔记本ubuntu
Codex-鼎好主机4090
Codex-实验服务器01
```

同一个机器人有两类会话行为：

### 私聊：机器级控制面

私聊不绑定某个持久 Codex thread，用于：

- 查看本机最近受管对话
- 新建 Codex 对话和对应飞书群
- 查看 Codex 额度、桥状态和长连接状态
- 修改私聊临时任务的模型、速度和权限
- 执行与已有对话无关的一次性任务
- 接收断网、配置、兼容性和恢复告警

### 群聊：持久 Codex 对话

每个受管 Codex thread 对应一个飞书群。桥使用数据库中的 `thread_id <-> chat_id` 绑定路由，而不是依赖可修改的群名。群名默认是：

```text
<Codex 对话名称>-<机器名称>
```

群里的新消息成为该 thread 的新 turn；执行中的补充要求可以 steer 当前 turn 或按队列顺序等待。

## 中间进度与最终交付

桥不是“收到消息后长时间沉默”的简单 Webhook。它会：

- 收到消息后立刻发送入队确认或进度卡片
- 周期性投影 Codex commentary、计划和当前工具活动
- 及时转发需要用户处理的审批或选择
- 任务结束后发送最终文字
- 上传本轮专用 outbox 内通过检查的交付文件
- 飞书断网时把未发送内容保留在 SQLite outbox 并重试

如果用户先发送图片/视频/文件而没有文字，桥只暂存附件；同一用户在同一会话发出下一条真实任务文字后，附件与文字原子合并交给 Codex。`/status` 等桥控制命令不会消耗暂存附件。

## 当前信任模型

当前版本的目标用户是：

- 设备所有者
- 飞书组织和自建应用的管理员
- Codex 账号的合法使用者
- 唯一会向机器人下发执行指令的人

项目通过 owner 配对、租户校验、应用隔离、群绑定和本地权限限制减少误操作，但没有被设计成允许互不信任的多用户共享执行环境。

## 非目标

当前项目不试图：

- 为不受信任公众提供任意代码执行服务
- 用一个 App ID 同时直连多台独立执行机器
- 把本地 Codex thread 自动同步为跨机器共享历史
- 替代 Codex 官方认证、沙箱或额度系统
- 保证模型生成的所有命令都安全
- 在没有明确授权的情况下复制用户凭据或私密数据

## 成功标准

一台机器完成部署的最低成功标准是：

1. 只有该机器的机器人消费该应用事件。
2. 未配对用户不能执行命令。
3. 私聊控制命令正常。
4. 群消息进入正确 thread。
5. 中间进度和最终答复都可回传。
6. 图片/文件收发和暂存逻辑正常。
7. 飞书断网与服务重启后消息不会静默丢失或重复执行。
8. 机器重启后服务恢复。
9. Codex CLI 版本变化会触发兼容门禁，而不是静默使用旧假设。
