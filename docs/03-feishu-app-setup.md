# 创建并发布飞书应用

本流程为每台目标机器创建一个独立的飞书企业自建应用。示例机器名使用 `鼎好主机4090`，对应机器人名：

```text
Codex-鼎好主机4090
```

## 准备条件

- 已登录飞书开放平台。
- 当前组织是“因时而得谷”，而不是个人测试组织或其他企业。
- 当前账号能创建企业自建应用并提交/批准发布。
- 已确定唯一机器名称。
- 目标主机能访问飞书开放平台和 OpenAI/Codex 服务。

## 第 1 步：创建企业自建应用

1. 打开飞书开放平台开发者后台。
2. 确认右上角组织为“因时而得谷”。
3. 点击“创建企业自建应用 / Create Custom App”。
4. 不要选择商店应用；Agent-ready 模板也不是必需条件。
5. 填写：

   - 名称：`Codex-鼎好主机4090`
   - 描述：`远程监督和操控鼎好4090主机上的 Codex`
   - 图标：建议所有机器使用同一图形、不同名称

6. 创建后确认页面显示的是正式应用，并归属于因时组织。

## 第 2 步：记录 App ID，保护 App Secret

进入：

```text
基础信息
└── 凭证与基础信息
```

- App ID 不是密码，可填写进目标主机 `config.toml`。
- App Secret 是长期应用凭据，只能在目标主机本地使用。
- 不要把 Secret 发到飞书、Codex、Issue、Pull Request、截图或日志。
- 不要把 Secret 写进本仓库。

目标配置：

```toml
[feishu.conversation]
app_id = "你的 App ID"
app_secret_env = "FEISHU_CONVERSATION_APP_SECRET"
```

Secret 文件：

```text
FEISHU_CONVERSATION_APP_SECRET='只在目标机器填写'
```

## 第 3 步：添加机器人能力

进入：

```text
添加应用能力 / Add Features
└── Bot / 机器人
└── 添加
```

添加的是“应用机器人”，不是某个群里的 Webhook 自定义机器人。如果页面提供“接收群聊所有消息/无需 @”选项，请启用。

## 第 4 步：开通最小权限

进入：

```text
开发配置
└── 权限管理 / Permissions & Scopes
└── 开通权限
```

在“应用身份”下按稳定权限标识搜索并开通：

| 权限标识 | 用途 |
|---|---|
| `im:message.p2p_msg:readonly` | 接收 owner 私聊 |
| `im:message.group_msg` | 接收对话群消息，不限于 @ |
| `im:message:readonly` | 历史补扫、读取消息资源 |
| `im:message:send_as_bot` | 以机器人身份发送消息 |
| `im:message:update` | 更新中间进度卡片 |
| `im:resource` | 上传和下载图片、文件、媒体 |
| `im:chat:create` | 为 Codex thread 创建飞书群 |
| `im:chat:read` | 查找和恢复已创建群 |
| `im:chat:update` | 把工作目录和对话开始时间写入已有群描述 |
| `im:chat:operate_as_owner` | 管理机器人创建的群 |

当前桥不需要通讯录、云文档、日历、审批或多维表格权限。若以后新增飞书文档报告能力，应单独设计并只增加对应权限。

权限可能显示“待发布”，这是正常状态；创建版本后才正式生效。

## 第 5 步：让目标桥建立初始长连接

飞书控制台可能要求应用已有在线长连接才能保存订阅方式。目标主机在已经安装桥并填写 App ID/Secret 后运行：

```bash
cd /path/to/Yinshi/codex-feishu-bridge
set -a
source ~/.config/codex-feishu-bridge/secrets.env
set +a
.venv/bin/codex-feishu-bridge run
```

保持终端运行。如果控制台允许直接保存长连接配置，可以先完成第 6/7 步；遇到“应用未建立长连接”时再回来执行本步骤。

## 第 6 步：订阅消息事件

进入：

```text
开发配置
└── 事件与回调
└── 事件配置
```

1. 订阅方式选择“使用长连接接收事件”。
2. 保存订阅方式。
3. 点击“添加事件”。
4. 添加应用身份事件：

   ```text
   im.message.receive_v1
   ```

5. 不要同时添加同一消息事件的旧版本。

## 第 7 步：订阅卡片回调

进入同一页面的“回调配置”：

1. 选择“使用长连接接收回调”。
2. 点击“添加回调”。
3. 添加新版“卡片回传交互”：

   ```text
   card.action.trigger
   ```

4. 保存。

该回调用于模型、权限、CLI 兼容、审批等按钮。不要选择只支持旧 Webhook 的旧版卡片回调。

## 第 8 步：安全设置和可用范围

- 当前架构使用长连接，不需要公网回调 URL、域名白名单或端口映射。
- 可用范围至少包含实际 owner；单人组织可选全部成员。
- 不要把机器人开放给无权操作目标机器的人。
- 不要把该应用安装到不受信任的外部群。

## 第 9 步：发布 1.0.0

进入：

```text
应用发布
└── 版本管理与发布
└── 创建版本
```

填写：

- 版本号：`1.0.0`
- 更新说明：

  ```text
  首次发布 Codex-鼎好主机4090：
  支持私聊控制、Codex 对话群、中间进度卡片和文件传输。
  ```

- 可用范围：包含 owner

确认版本包含机器人能力、全部权限、消息事件和卡片回调，然后保存并申请发布。如果显示待管理员审核，由因时管理员在管理后台或飞书审批中批准。

以后修改权限、事件、回调、机器人信息或可用范围，都应创建新版本并重新发布。

## 第 10 步：客户端可见性检查

发布后在飞书搜索 `Codex-鼎好主机4090`：

- 能打开机器人私聊
- 私聊有输入框
- 机器人名称和图标正确
- owner 在可用范围

没有输入框时优先检查：

1. 是否添加机器人能力
2. 是否开通 `im:message.p2p_msg:readonly`
3. 是否添加 `im.message.receive_v1`
4. 是否发布包含这些变更的新版本
5. owner 是否在可用范围

## 第 11 步：配对

目标主机保持桥前台运行，另开终端生成配对码：

```bash
set -a
source ~/.config/codex-feishu-bridge/secrets.env
set +a
.venv/bin/codex-feishu-bridge pair-code
```

在该机器人私聊发送输出的：

```text
配对 XXXXXXXX-XXXXXXXX
```

收到“Codex 应用已与本机安全配对”后，继续执行 bootstrap 和验收。配对码不得发送给其他账号。

## 常见错误

### 保存长连接时提示应用不在线

先在目标主机配置 App ID/Secret 并运行桥前台进程，再回控制台保存。

### 发布后仍没有新权限

检查是否创建并发布了包含最新配置的版本，以及管理员审核是否完成。

### 机器人能发送但不能接收

检查消息事件、单聊/群聊权限、可用范围和飞书事件日志。

### 卡片显示但按钮没有结果

检查新版 `card.action.trigger`、长连接回调配置和发布版本。

### 同一应用在两台机器运行

立即停止其中一个实例。飞书会随机把事件交给任一 client，无法保证命令落到目标机器。为第二台机器创建独立应用。
