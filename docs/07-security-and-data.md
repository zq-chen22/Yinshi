# 安全与数据管理

## 信任模型

当前实现面向“一个自然人管理自己拥有的飞书组织和计算机”。配对后的 owner 发出的消息可触发本机 Codex，因此机器人会继承 Codex 被授予的文件、命令和网络能力。

项目当前不适合作为多租户执行平台，也没有实现团队成员之间的细粒度 RBAC、审计审批链或资源配额隔离。将机器人开放给其他人之前必须重新做威胁模型。

## 需要保护的资产

- 飞书 App Secret；
- Codex 登录和账号令牌；
- owner 的飞书身份及配对记录；
- Codex thread 内容与 rollout；
- 本地项目源码、数据集和工作区；
- SQLite 中的消息、绑定、去重、审批和发件箱元数据；
- 用户上传到 inbox 的图片、音视频和文件；
- Codex 生成到 outbox 的交付物；
- systemd 日志和错误栈中可能出现的文件名、任务摘要。

## 凭据规则

App ID 可以写入配置；App Secret 只能通过本机环境变量加载。推荐位置：

```text
~/.config/codex-feishu-bridge/secrets.env
```

要求：

```bash
chmod 600 ~/.config/codex-feishu-bridge/secrets.env
stat -c '%a %U %G %n' ~/.config/codex-feishu-bridge/secrets.env
```

禁止把真实 Secret 放入：

- `config.toml` 或源码；
- Git 提交、分支、Issue、PR、Release 附件；
- 飞书/Codex 聊天；
- 屏幕截图、录屏和公开日志；
- 未加密的云盘同步或备份。

怀疑泄露时，立即在飞书开放平台重置 Secret、更新本机文件、重启服务并审查仓库历史。仅从当前文件删除不足以清除 Git 历史。

## 身份与授权

- 配对码只能在机器人 P2P 私聊中使用，且短时有效；
- 首次配对记录该飞书应用作用域内的 owner 身份；
- 后续命令校验 sender、tenant、chat 类型与 thread/chat 绑定；
- 群聊未绑定时不得自动猜测目标 thread；
- 卡片按钮还要经过与文本命令相同的 owner 和会话校验；
- 每台机器使用独立飞书应用，避免同一应用多实例随机消费消息。

飞书的 `open_id` 对不同应用可能不同，不能手工从另一个应用复制。新机器应使用本机桥生成的配对码重新配对。

## Codex 权限预设

| 飞书显示 | approval policy | sandbox | 适用场景 |
| --- | --- | --- | --- |
| Read Only | `on-request` | `read-only` | 审阅、调查、状态检查 |
| Default | `on-request` | `workspace-write` | 一般开发和文件修改 |
| Full Access / YOLO | `never` | `danger-full-access` | 只有单一所有者明确接受全部风险时 |

公开仓库默认 Default。YOLO 意味着来自飞书的自然语言可能在当前用户权限范围内读写广泛文件、执行命令并进行网络操作，错误理解、提示注入、恶意附件或账号失窃都可能造成实际损害。“机器只有我用”能降低其他人员误用，不会消除模型错误、供应链、凭据和远程账号风险。

建议：

- 日常使用 Default；
- 高风险任务前先备份并缩小 `allowed_workspace_roots`；
- 对删除、发布、付款、外发、修改权限等动作保留明确人工确认；
- 不把 root 密码写进配置或聊天；
- 服务以普通用户运行，sudo 仍要求独立授权。

## 消息、附件和外发

用户先发送的无文字图片、视频、音频或文件会被下载到隔离 inbox 并暂存，直到同一用户在同一会话发送普通文字。桥控制命令不会消耗附件。

下载默认上限为 50 MiB，上传默认上限为 30 MiB；具体飞书媒体类型可能有更低上限。大小限制只能控制资源消耗，不能证明文件安全。不要自动执行未知二进制、宏文档或脚本。

任意本机路径不会因为出现在 Codex 最终文字中就自动外发。自动回传仅处理本轮专用 outbox 内新生成、通过秘密检查且未超过上限的文件。这个边界用于减少误传已有文件和凭据，但不能替代对生成内容本身的审查。

## 本地数据目录

默认目录：

```text
~/.local/share/codex-feishu-bridge/
├── bridge.sqlite
├── inbox/
├── outbox/
├── admin-scratch/
├── workspaces/
└── service.lock
```

目录应为 `0700`，文件默认受服务 `UMask=0077` 保护。不要把状态目录放进 Git 仓库或多人共享目录。

当前项目没有自动数据保留策略。管理员应根据磁盘空间和隐私要求定期清理已完成、无需保存的 inbox/outbox/scratch 内容；清理前停止服务或确保条目不再被发送队列引用。数据库记录的保留与清理功能仍属于未来工作。

## 日志

代码不应主动打印 App Secret，但任务文本、thread ID、chat ID、文件路径和错误响应仍可能具有敏感性。限制 journal 访问权限，设置合理持久化和轮转策略；提交 Issue 前先脱敏。

安全查看：

```bash
journalctl --user -u codex-feishu-bridge.service --since today --no-pager
```

分享日志前删除身份 ID、消息 ID、绝对路径、任务内容和访问令牌。不要用简单的全量复制粘贴替代脱敏。

## 备份、轮换与退役

备份必须加密，Secret 与普通状态最好分开保存。恢复到新机器时不要同时运行旧机和新机的同一 App ID；先停止旧机、确认长连接下线，再迁移或更换应用。

主机退役流程：

1. 停止并禁用 systemd 服务；
2. 在飞书开放平台停用应用或重置 Secret；
3. 撤销或清理 Codex 本地登录；
4. 加密归档确有保留义务的数据；
5. 安全删除本地 Secret、数据库、inbox/outbox 和工作区；
6. 清理用户服务文件和 linger；
7. 在因时组织移除或标记旧机器人与群。

## 安全事件最低响应

- 飞书账号异常：冻结会话/设备、停止桥、检查机器人可见历史；
- App Secret 泄露：立即重置并检查应用日志与权限；
- Codex 凭据泄露：按 OpenAI 账号流程撤销会话并重新登录；
- 疑似远程恶意执行：断开桥网络或停止服务，保留只读证据，再检查进程、日志、工作树和外部副作用；
- 仓库误提交秘密：轮换秘密优先，然后重写历史并通知所有克隆者。
